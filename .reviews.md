# vm.py 深度审查报告

## 严重问题（Critical）

### 1. `pop()` 栈下溢时静默返回 None
- **位置**: `vm.py:61-67`
- **描述**: `pop()` 方法在栈为空时返回 `None` 而不是抛出异常。这会导致后续算术运算（如 `ADD`、`SUB`）中 `None + int` 引发类型错误，使错误定位极其困难。栈下溢是 VM 的核心安全边界，必须显式抛出。
- **修复建议**: 栈为空时抛出 `StackUnderflowError("cannot pop from empty stack")`，调用方在需要容错的场景自行捕获。

### 2. `DIV` 指令未处理除零错误
- **位置**: `vm.py:118-122`
- **描述**: `DIV` 操作码直接执行 `a / b`，当 `b == 0` 时会抛出原生 `ZeroDivisionError` 导致 VM 崩溃。同时整数除法会返回 `float`，破坏栈数据类型一致性。
- **修复建议**: 在除法前检查 `b == 0`，抛出专用 `DivisionByZeroError`；明确整数除法使用 `//` 还是 `/`，并在文档中说明。

### 3. `LOAD` 指令访问不存在变量时抛出 KeyError
- **位置**: `vm.py:133-137`
- **描述**: `LOAD` 指令直接通过 `self.variables[name]` 访问变量，变量不存在时抛出未受控制的 `KeyError`，导致 VM 意外终止而非优雅报错。
- **修复建议**: 使用 `self.variables.get(name)` 并在变量不存在时抛出 `VMError(f"undefined variable: {name}")`，保持错误体系一致。

### 4. 未知操作码被静默忽略
- **位置**: `vm.py:167-169`
- **描述**: `step()` 方法中无法识别的操作码走到 `else` 分支，仅执行 `pass`，PC 仍自增但不做任何处理。这会导致程序损坏时静默跳过错误指令，产生难以调试的行为。
- **修复建议**: 抛出 `VMError(f"unknown opcode: {hex(opcode)} at pc={self.pc-1}")`，让程序错误及早暴露。

### 5. `peek()` 无偏移量边界检查
- **位置**: `vm.py:69-73`
- **描述**: `peek(offset)` 未校验 `offset` 的合法性，负数或过大的偏移量会导致 `IndexError` 或返回错误位置的数据，被 `DUP` 等指令调用时可能造成栈状态损坏。
- **修复建议**: 增加 `if offset < 0 or offset >= len(self.stack)` 校验，抛出 `StackUnderflowError`。

## 中等问题（Medium）

### 1. `reset()` 未清空变量和内存
- **位置**: `vm.py:171-179`
- **描述**: `reset()` 方法只重置了栈、PC、SP、FP 和 running 标志，但 `self.variables` 和 `self.memory` 的数据残留到下一次运行，可能造成跨程序的数据泄露和逻辑错误。
- **修复建议**: 添加 `self.variables.clear()` 和 `self.memory = [0] * len(self.memory)`。

### 2. `RET` 指令未实现但静默通过
- **位置**: `vm.py:162-164`
- **描述**: `RET` 操作码只有 `pass`，既没有实现也没有抛出 `NotImplementedError`。与 `CALL` 的处理不一致，程序执行到 `RET` 时会静默跳过，导致控制流错误。
- **修复建议**: 统一处理方式，要么抛出 `NotImplementedError`，要么注释说明预留操作码。

### 3. `dump_memory()` 默认参数 end=0 导致无输出
- **位置**: `vm.py:191-200`
- **描述**: `dump_memory()` 的默认参数 `end=0` 使得无参调用时 `range(0, 0)` 为空，不输出任何内存内容，与用户直觉相悖。方法名暗示会"dump memory"，但实际什么都不做。
- **修复建议**: 将默认值改为 `end=None`，在方法内 `if end is None: end = len(self.memory)`。

### 4. `run()` 始终返回 None
- **位置**: `vm.py:80-88`
- **描述**: `run()` 方法签名标注返回 `Any`，但始终返回 `None`。注释中也写着"TODO: return top of stack as result?"，说明功能未完成。调用方无法获取程序执行结果。
- **修复建议**: 返回 `self.peek()`（栈顶值）作为程序退出码/结果，或明确返回 `None` 并更新文档。

### 5. `JMP` 指令无地址边界校验
- **位置**: `vm.py:143-145`
- **描述**: `JMP` 直接将 `pc` 设置为操作数，不校验地址是否在程序范围内。跳转到非法地址后，`step()` 中 `pc >= len(program)` 才会停止，但这是隐式行为而非显式错误。
- **修复建议**: 在跳转前检查 `0 <= addr < len(self.program)`，越界时抛出 `VMError(f"jump to invalid address: {addr}")`。

## 轻微问题（Minor）

### 1. `__init__` 末尾存在无意义 `pass`
- **位置**: `vm.py:54`
- **描述**: `__init__` 方法末尾单独一行 `pass`，上方已有两条 TODO/FIXME 注释，`pass` 没有实际作用，属于冗余代码。
- **修复建议**: 删除 `pass`，保留 TODO 注释即可。

### 2. 算术运算缺少操作数类型检查
- **位置**: `vm.py:105-122`
- **描述**: `ADD`、`SUB`、`MUL`、`DIV` 等指令直接对弹出的值进行算术运算，若栈上压入了非数字类型（如字符串），会抛出 Python 原生类型错误而非 VM 体系内错误。
- **修复建议**: 增加 `isinstance(a, (int, float))` 检查，类型不匹配时抛出 `VMError`。

### 3. `main()` CLI 入口未实现文件读取
- **位置**: `vm.py:215-222`
- **描述**: `main()` 函数检查了命令行参数并打印用法，但参数存在时直接 `return 0`，既不打开文件也不执行程序，功能完全缺失。
- **修复建议**: 实现文件读取和程序加载逻辑，或暂时抛出 `NotImplementedError`。

### 4. `CALL` 指令直接抛出 NotImplementedError 中断运行
- **位置**: `vm.py:159-161`
- **描述**: `CALL` 指令遇到就抛异常，没有 try/except 包裹，整个 `run()` 会因此中断。虽然功能未实现，但作为 VM 核心指令，至少应该在文档中明确标注不支持。
- **修复建议**: 在 VM 类 docstring 中列出已实现/未实现指令清单；或在 `run()` 外层捕获 `NotImplementedError` 并给出友好提示。

### 5. `StackUnderflowError` 定义了但从未使用
- **位置**: `vm.py:15-17`
- **描述**: 定义了专门的 `StackUnderflowError` 异常类，但整个代码中没有任何地方抛出它。`pop()` 栈下溢时返回 `None`，`peek()` 也未使用。
- **修复建议**: 在 `pop()`、`peek()` 中实际使用 `StackUnderflowError`，保持异常体系的完整性。
